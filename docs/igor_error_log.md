## EDCEdgeWidth 开发错误回顾

### 1. 把 subwindow 当普通 window 用

* 误把 `panel#subwindow` 路径当作普通窗口名处理。
* 典型问题是对子图窗乱用 `DoWindow/T`、类似顶层窗口的操作。
* 结论：panel 内 graph 必须按 **subwindow** 逻辑管理。

### 2. 结果索引差点按 list row 写

* 一开始容易把 listbox 的行号当成真实结果索引。
* 但正确映射应是：
  `edc_show_i -> edc_xrise[i], edc_xfall[i], edc_width[i]`
* 结论：结果必须按 wave 名编号写，不能按显示顺序写。

### 3. 结果 wave 长度最初定义不对

* 早期按“扫描到的 wave 个数”建结果 wave。
* 这样如果 `edc_show_*` 缺号，就会越界或错位。
* 结论：结果 wave 长度必须按 **最大编号 + 1** 建立。

### 4. Source DF 输入框只改了显示，没真正写回变量

* 一开始手动输入路径后，程序仍在用旧路径扫描。
* 根因是 `SetVariable` 回调里没有把 `sva.sval` 写回全局字符串变量。
* 结论：字符串输入控件要显式写回 `SVAR`。

### 5. SetVariable 的字符串绑定理解有偏差

* 底部显示一度显示成变量路径本身，而不是变量内容。
* 说明字符串绑定和刷新逻辑没处理好。
* 结论：Igor 里字符串型 `SetVariable` 要特别注意绑定方式和回调更新。

### 6. 错把 cursor 逻辑混进测量思路

* 明明已经有 `RiseX1/RiseX2/FallX1/FallX2` 四个窗口输入，还试图让 cursor 参与图形状态。
* 会让逻辑变乱，也容易残留旧状态。
* 结论：测量应只由窗口参数驱动，cursor 不是必要状态源。

### 7. 图上标记线的 Draw 逻辑踩坑

* `DrawAction regroup=...` 这种思路不对。
* 正确做法是用 `gstart/gstop` 建组，再统一删除旧组。
* 结论：Igor 的 draw object 分组必须按它自己的机制来。

### 8. 用含 `x` 的表达式给普通变量赋值

* 比如把 `x` 直接写在普通 `Variable` 赋值里。
* 会报 `P or X used outside of a wave assignment loop`。
* 结论：`x/p` 只能出现在 **wave assignment** 语境中。

### 9. UI 追加控件时发生重复定义

* 后期加 `Auto Rise / Auto Fall` 时，曾经在 panel 里重复放置同名按钮。
* 结论：每次加新控件都要检查是否与旧布局重复。

### 10. 手动改窗口后图没有立即刷新

* 一开始只改了数值，没有同步刷新图上的 window 边界线。
* 结论：参数输入、图形标记、结果显示三者要保持联动。

---

## 最后总结

这次最核心的经验有四条：

1. **Igor 的窗口系统和普通 GUI 框架不一样，subwindow 要单独对待。**
2. **结果存储一定要先想清楚索引规则，再写代码。**
3. **字符串控件和数值控件在 Igor 里不是同一种处理方式。**
4. **先把数据流理顺：输入窗口 → 测量 → 写回结果 → 图上刷新。**
